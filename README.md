## 🚀 Getting Started

To spin up the PitchMind platform locally, follow these streamlined development setup steps:

### 1. Backend API Server Setup
1. Navigate to the backend directory:
   ```bash
   cd backend
   ```
2. Create and activate a Python virtual environment:
   ```bash
   python -m venv .venv
   .venv\Scripts\activate  # Windows
   source .venv/bin/activate  # macOS/Linux
   ```
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Create a `.env` file with backend settings:
   ```env
   DATABASE_URL=sqlite:///./pitchmind.db
   SECRET_KEY=super-secret-jwt-signing-key-change-in-production
   ALGORITHM=HS256
   PITCHMIND_OTP_BYPASS_ENABLED=False
   ```
5. Run the FastAPI development server:
   ```bash
   uvicorn app.main:app --reload --port 8000
   ```
   *The API will be live at `http://127.0.0.1:8000/docs`.*

### 2. Frontend Next.js Client Setup
1. Navigate to the frontend directory:
   ```bash
   cd frontend
   ```
2. Install npm packages:
   ```bash
   npm install
   ```
3. Create a `.env.local` configuration:
   ```env
   NEXT_PUBLIC_API_URL=http://localhost:8000
   # For Supabase integration (optional, automatically switches to local fallback if empty):
   NEXT_PUBLIC_SUPABASE_URL=
   NEXT_PUBLIC_SUPABASE_ANON_KEY=
   ```
4. Run the Next.js development server:
   ```bash
   npm run dev
   ```
   *The web portal will be accessible at `http://localhost:3000`.*

## 🐳 Containerization & Deployment

PitchMind is fully containerized with multi-stage Docker builds optimized for size, performance, and security.

### Docker Compose Quickstart
To spin up both the FastAPI backend and Next.js frontend in isolated production containers using a single command:
```bash
docker-compose up --build
```
* **Frontend:** Exposed on `http://localhost:3000`
* **Backend:** Exposed on `http://localhost:8000`

## 🔒 Enterprise-Grade Security Refinement

During recent core refinements, the authentication and observability layers were significantly hardened:
1. **JWT-Signed Authorization Flow**: Replaced vulnerable mock ID tokens with cryptographically signed JSON Web Tokens (JWT) using the `HS256` or `RS256` signature algorithms.
2. **Dynamic Supabase Hybrid client**: Built-in support to auto-detect Supabase credentials. If present, it enables cloud authentication; otherwise, it handles it gracefully using a secure native REST fallback.
3. **Structured Custom Logging**: Outdated console prints are replaced by an industry-standard structured logger, logging crucial database, lifecycle, and validation events with JSON-like structure.
4. **Dev Bypass Blocked**: Insecure test OTP bypass codes are permanently locked down in production environments.
